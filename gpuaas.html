Bare Metal GPU Servers

Dedicated physical hardware — no hypervisor overhead, no noisy neighbours. Full GPU PCIe or SXM lane allocation. Ideal for max-performance training and latency-critical inference.

FAQ

Common Questions from GPU Engineers

What is the difference between bare metal GPU and a GPU VM? When should I use each? ▾

Bare metal GPU means a physical server with no hypervisor — you get 100% of the GPU's PCIe bandwidth, NVLink capacity, and memory. There is no "hypervisor tax" stealing 5–15% of your GPU's throughput. Use bare metal for: LLM pre-training, large-scale fine-tuning, multi-GPU NVLink topologies, and workloads where every TFLOP matters. GPU VMs use hypervisor passthrough (SR-IOV) or NVIDIA vGPU virtualisation to partition one or more physical GPUs across multiple tenants or VMs. They're faster to provision, cheaper per hour for dev/test workloads, and easier to snapshot and clone. Use VMs for: development environments, CI/CD testing, smaller inference workloads, and multi-tenant team environments where you need isolation without dedicated hardware costs. Thynkwise provisions both and can combine them — bare metal clusters for production training with VM-based dev environments pointing at the same model checkpoints.

How do I choose between NVIDIA H100/H200 and AMD MI300X for LLM training? ▾

Both are world-class AI training GPUs but with different strengths. NVIDIA H100/H200 has the most mature software ecosystem — CUDA, cuDNN, NCCL, TensorRT, Megatron-LM, and every major framework are battle-tested on Hopper. If your team uses standard PyTorch/CUDA stacks and you need maximum software compatibility, H100/H200 is the lower-friction choice. AMD MI300X has a critical advantage: 192 GB HBM3 memory (vs H100's 80 GB, H200's 141 GB). If you're running models that don't fit on a single H100 — Llama 3.1 405B, DeepSeek 671B, or any model over 100B parameters at FP16 — MI300X can host them on a single GPU where H100 requires tensor parallelism across 2–4 GPUs. This simplifies deployment significantly. AMD ROCm has improved substantially but NVIDIA's CUDA ecosystem remains more mature. For teams willing to invest in ROCm compatibility (which many popular frameworks support natively), MI300X's memory advantage makes it compelling for very large model inference. Thynkwise runs both and helps you benchmark on your specific model before committing to a hardware choice.

What does NVLink actually do and why does it matter for distributed training? ▾

NVLink is NVIDIA's proprietary GPU-to-GPU interconnect that runs at 900 GB/s per GPU (NVLink 4.0 on H100) to 1.8 TB/s per GPU (NVLink 5.0 on B200). Compare this to PCIe 4.0 x16, which only delivers ~64 GB/s bidirectional. Why does this matter for training? Distributed training across multiple GPUs requires constant gradient synchronisation — all-reduce operations that average gradients across every GPU so the model parameters stay synchronised. On a PCIe system, this all-reduce communication bottlenecks training because the interconnect can't keep up with compute. On an NVLink system, gradient synchronisation is so fast it becomes negligible. For 8-GPU nodes, NVLink means you can run tensor parallelism (splitting a single transformer layer across GPUs) efficiently — this is impossible at useful speeds over PCIe. When you go beyond one node, you add InfiniBand NDR at 400 Gb/s between nodes. The combination of NVLink within a node and InfiniBand between nodes is what makes multi-node LLM training at 64–512+ GPU scale feasible. Thynkwise configures the entire NVLink + InfiniBand topology and NCCL collective operations for your specific parallelism strategy.

What is NVIDIA MIG and when should I use it? ▾

Multi-Instance GPU (MIG) is an NVIDIA feature available on A100 and H100 GPUs that partitions a single physical GPU into up to 7 isolated GPU instances, each with dedicated memory, compute, and bandwidth. Each MIG instance is fully isolated — faults in one instance don't affect others, and there's no cross-contamination of data or performance. Use MIG when: you have multiple small workloads that don't each need a full GPU, you need strong tenant isolation in multi-user environments (important for regulated industries), you want to maximise GPU utilisation (running 7 small inference workloads on one H100 instead of provisioning 7 separate GPUs), or you're building a development environment where different teams need isolated GPU access. MIG is not appropriate for large model training that needs the full GPU's memory and compute, NVLink-dependent workloads (MIG partitions don't share NVLink), or workloads requiring more VRAM than the MIG partition size provides. Thynkwise configures MIG profiles to match your specific workload mix and manages partition reconfiguration as demand patterns change.
